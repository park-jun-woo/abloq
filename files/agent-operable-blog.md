## 0. 모델 요약

```
사람: 인사이트 (주제·주장·관점의 결정 — 유일한 인간 입력)
  ↓
[에이전트 퀘스트]  집필 · 번역 · 갱신 · 근거 보강 · 링크 큐레이션  (비결정 — 산문을 만지는 일만)
[코드 게이트]      구조·근거·정책 판정 (reins, 커밋/빌드 시)      (2026-06-10 720/720 PASS 실증)
[운용 백엔드]      파생물 생성 · 아카이브 · 스캔 · 측정 · 큐       (결정적 — 스케줄/이벤트 실행)
  ↓
백엔드의 우선순위 큐(크롤 히트·GSC·신선도·근거·클러스터) → 다음 에이전트 퀘스트의 입력 (래칫)
```

- **분업 원칙:** 에이전트는 산문을 만지는 비결정 노동에만 쓴다. 생성·스캔·측정·외부 API 호출·판정은 전부 결정적 코드(게이트 + 운용 백엔드 §3.3)다. 코드로 되는 일에 에이전트를 쓰면 비용·비결정성·치즈 공격면만 늘어난다.

- **사용자가 얻는 것:** 블로그가 아니라 **AI 가시성**. "ChatGPT/Claude/Perplexity가 내 콘텐츠를 인용하게 만들고, 그 상태를 유지해 주는 운용."
- **차별점:** GEO 대행사·SaaS는 체크리스트와 대시보드(사람 노동)를 판다. 이 프레임워크는 **노동을 에이전트가 하고 판정을 게이트가 하는 박스**를 무료로 공개한다. *"GEO 대행사를 박스에 담아 오픈소스로 풀었다."*

## 1. 왜 지금, 왜 우리인가

**시장.** 전통 검색 -25% (Gartner, 2026), 미국 인구 31.3%가 생성형 AI 검색 사용. "AI에 인용되고 싶다"는 수요는 전 기업·창작자 공통인데, 공급은 컨설팅(비싸고 사람)과 SaaS 대시보드(보여주기만 함)뿐. **실행까지 해주는 도구가 없다.**

**우리의 비대칭.**

1. **실증 코퍼스:** parkjunwoo.com 12개 언어 × 60편 = 720편이 이 파이프라인으로 실제 운용 중. 2026-06-10 섹션 구조 정규화 퀘스트 720/720 PASS(FAIL 0)가 첫 공개 가능 사례.
2. **기계 판정 스택:** reins 게이트·퀘스트 CLI·치즈 방어 룰셋이 이미 있다. GEO 원칙은 거의 전부 기계 판정 가능(§5)이라 이 스택에 그대로 올라간다.
3. **인프라 자산:** .md 직접 서빙(generate-md.sh — AI 컨텍스트용 노이즈 제로 포맷, [[AI-SEO-전략]]의 핵심 실행이 이미 됨), 12언어 hreflang sitemap, IndexNow, CloudFront 로그 파이프라인(deploy/logs + analyze-stats.py), GSC API 연동.

**목적 함수가 생겼다는 것.** "에이전트가 블로그를 운영한다"만으로는 운영의 좋고 나쁨을 판정할 수 없었다. GEO가 측정 가능한 목적 함수(AI 인용·크롤 히트·GSC 노출)를 주고, 측정이 가능하면 래칫이 걸리고, 래칫이 걸리면 퀘스트가 된다. 루프가 닫힌다.

## 2. 라이선스 — 전부 오픈

전체가 MIT 무료 오픈소스다. 프레임워크 본체(Hugo 템플릿 + 레이아웃 + blog.yaml SSOT + 배포 IaC), 게이트 룰셋(GEO 게이트 Go 패키지, §5), 운용 백엔드(§3.3), 에이전트 퀘스트 팩(§4 — 게이트+태스크 트리+컨텍스트) 모두 같은 라이선스로 공개한다.

비개발자 셀프서비스 발행(폐기된 CMS의 부활 시나리오)은 Phase 3에서 이 프레임워크 + 저작 도구 조합으로 자연 해소된다 — CMS를 다시 짓는 게 아니라 퀘스트가 CMS를 대체.

## 3. 아키텍처

### 3.1 SSOT: blog.yaml 한 장

일반화 비용을 통제하는 핵심. parkjunwoo.com의 하드코딩(12언어·5섹션·개인 CSS)을 전부 일반화하지 않는다. **설정 한 장에서 파생되는 것만 일반화한다.**

```yaml
# blog.yaml — 블로그 하나의 전 선언 (yongol의 manifest 사상)
site:        { baseURL, title, author }          # → hugo.toml 생성
languages:   [ko, en, ja, ...]                   # 기본 언어 + 번역 대상 (1~N개)
sections:    [opinion, tech, ...]                # 디렉토리 기반 섹션
structure:                                       # 글의 정규 구조 = 게이트 룰의 입력
  order: [image, attribution, body, related, further, sources, changelog]
  headings: { related: {ko: "관련 글", en: "Related", ...} }   # 언어별 헤딩 맵
geo:
  crawlers:  { training: allow|block, search: allow, fetch: allow, bytespider: block }
  llms_txt:  auto                                # 발행 글 목록에서 자동 생성
  jsonld:    [Article, Person]
  freshness_days: 90                             # 신선도 퀘스트 임계
  min_sources: 1                                 # 근거 게이트 임계
  min_internal_links: 2                          # 클러스터 게이트 임계
deploy:      { provider: s3-cloudfront, terraform: true, indexnow: true }
```

여기서 파생: hugo.toml(다국어) · robots.txt · llms.txt 템플릿 · sitemap(hreflang) · JSON-LD 파셜 · 게이트 룰 파라미터 · 퀘스트 CLI 설정. **blog.yaml이 바뀌지 않는 한 어떤 글도 게이트를 우회할 수 없다** — 제약은 계약.

### 3.2 저장소 형태

테마(서브모듈) ❌ — blog.yaml에서 파생되는 파일(hugo.toml·robots.txt·llms.txt 등)이 테마 경계를 넘고, 사용자 수정과 테마 업데이트가 충돌한다. **템플릿 저장소(degit-식 복제) + 도구 바이너리** ⭕.

```
my-blog/
  blog.yaml                  # SSOT
  CLAUDE.md                  # 에이전트 운영 매뉴얼 (생성됨 — 게시 절차 = 퀘스트 호출)
  content/{lang}/{section}/  # 글
  layouts/ assets/ static/   # 프레임워크 제공 (수정 가능, 게이트와 무관)
  quests/                    # 설치된 퀘스트 팩
  quests/queue/              # 운용 백엔드가 떨어뜨린 작업 큐 (에이전트가 소비)
  deploy/terraform/          # IaC (S3+CF+WAF + 백엔드, 옵션)
```

CLAUDE.md를 프레임워크가 생성한다는 게 포인트 — **에이전트가 1급 사용자**라는 선언이고, 설치 직후부터 Claude Code가 운영을 인수할 수 있다.

### 3.3 운용 백엔드 — 코드로 되는 일은 전부 여기로

초안(v0)은 GEO 인프라·아카이브·스캔·측정까지 전부 에이전트 퀘스트로 설계했는데, 이 중 산문을 만지지 않는 일은 전부 결정적 코드다. 에이전트로 돌리면 토큰 비용과 비결정성만 얹힌다. 그래서 **상시 실행 백엔드 서비스(yongol 기반 Go+Gin, 셀프호스트)**로 분리한다. 정적 사이트는 그대로 두고, 백엔드는 생성·측정·스케줄·영수증만 담당한다. 백엔드 자체를 yongol SSOT(manifest·OpenAPI·DDL·SSaC·Rego·Hurl)로 선언하고 코드는 생성한다 — 도메인 노동(파일 스캔·로그 파싱·외부 API)은 `func/` 커스텀 함수로 두고 CLI와 공유.

| 모듈 | 트리거 | 하는 일 | v0에서의 위치 |
|---|---|---|---|
| 파생물 생성기 | blog.yaml 변경·빌드 | hugo.toml · robots.txt · llms.txt · sitemap(hreflang) · JSON-LD 생성 + 일치 검증 | 구 "GEO 인프라 퀘스트" 전체 |
| 아카이버 | 배포 직후(webhook) | Wayback 저장 + IndexNow + GSC 색인 요청, 응답 영수증 저장 | 구 "아카이브 퀘스트" 전체 |
| 신선도 스캐너 | cron(월) | freshness_days 초과 글 검출 → 가시성 지표 가중 우선순위 큐 | 구 "신선도 퀘스트"의 검출 절반 |
| 주장-출처 스캐너 | cron(분기) | 무출처 수치 주장 검출(정규식+휴리스틱) + 기존 인용 URL 재검증(link rot) → 큐 | 구 "근거 보강 퀘스트"의 검출 절반 |
| 클러스터 스캐너 | cron(분기) | 고아 태그·고립 글·내부 링크 부족 검출 + 후보 링크 제안(태그 교집합 기반) → 큐 | 구 "클러스터 퀘스트"의 검출 절반 |
| 가시성 파이프라인 | cron(일/월) | §6 — CF 로그 집계 · GSC 폴링 · 인용 샘플링 실행·기록 · 월간 리포트 | 구 "가시성 리포트 퀘스트" 전체 |

- **상태:** PostgreSQL(yongol 표준 스택 — 영수증·시계열·큐). **자격증명:** CF 로그 버킷 RO · GSC SA · AI 엔진 API 키 — 전부 백엔드에만 있고 에이전트에는 없다.
- **큐 인터페이스:** `quests/queue/*.yaml`로 저장소에 커밋(또는 GitHub Issue 발행). 에이전트가 읽어서 소비하는, 백엔드→에이전트의 유일한 접점.
- **백엔드 없이도 동작:** 전 모듈이 CLI 서브커맨드로도 실행 가능(`abloq scan freshness` 등). 백엔드는 스케줄과 상태를 더한 것뿐 — 정적 사이트의 단순성을 해치지 않는다.

## 4. 에이전트 퀘스트 팩

산문을 만지는 비결정 노동만 퀘스트로 남긴다. 검출·생성·측정·외부 API는 §3.3 백엔드가 하고, 퀘스트 2~4는 백엔드가 떨어뜨린 큐를 소비한다. (v0의 GEO 인프라·아카이브·가시성 리포트 퀘스트는 백엔드 모듈로 이전 — 퀘스트에서 제외.)

| # | 퀘스트 | 트리거 | 게이트(핵심만) | 비고 |
|---|---|---|---|---|
| 0 | **집필** | 사람의 인사이트 명세 | 인사이트 명세(주장·논지 목록)의 각 항목이 본문에 대응(REVIEW) · 인용은 실재 검증(URL 200 + 메타 일치) · 출처 ≥ min_sources · 무출처 수치 주장 0 | 자료 수집·초안·퇴고를 에이전트가 수행 — 문체 품질은 비결정이라 게이트는 근거·논지 보존만 판정 |
| 1 | **번역** | 새 글 + 본문 실변경 | 구조 게이트 7룰(2026-06-10 실증) + 전 언어 slug 일치 + front matter 스키마 + 빌드 0 | 언어당 아이템 — N언어 = N아이템. 빌드·배포·아카이브는 코드(CI + 백엔드 아카이버) |
| 2 | **갱신** | 신선도 스캐너 큐 | 본문 실변경 + 구조 게이트 유지 · **본문 실변경 없는 lastmod 갱신은 차단**(honest-lastmod — 위조 역효과, 2026-06-10 720편 작업에서 내린 판단을 룰로 승격) | 인용의 76.4%가 30일 내 갱신 글 |
| 3 | **근거 보강** | 주장-출처 스캐너 큐 | 보강 후 글당 출처 ≥ min_sources · 신규 인용은 실재 검증(URL 200 + 메타 일치) · 큐에 없는 주장 변경 금지 | GEO 논문: 인용 +41% / 통계 +32% / 출처 +30% |
| 4 | **클러스터 큐레이션** | 클러스터 스캐너 큐 | 태그가 taxonomy SSOT에 존재 · 고아 태그 0 · 글당 내부 링크 ≥ min_internal_links · 고립 글(인링크 0) 0 | 백엔드가 후보 링크를 제안, 에이전트는 문맥에 맞는 선택과 앵커 문장 작성만 |

실행 형태: 0~1은 글마다, 2~4는 백엔드 큐 발생 시. **큐 소비 퀘스트가 본체다** — GEO는 상태가 아니라 운용이다.

치즈 방어 원칙(전 퀘스트 공통): front matter 보존 · 게이트 판정과 저장소 반영의 바이트 일치 · 큐 아이템 범위 밖 파일 변경 금지 · 본문 변경 퀘스트(2·3)는 기존 주장 삭제 금지(인사이트 명세 보존), 구조 변환 퀘스트(1)는 본문 무손실(multiset). 외부 부수효과(아카이브·색인)는 백엔드 영수증으로 이전 — **에이전트는 외부 API를 직접 치지 않는다.**

## 5. GEO 원칙 → 게이트 룰 변환표

GEO 담론은 컨설팅 언어로 떠돌지만, 실체는 거의 전부 기계 판정 가능하다. 이 변환이 이 프레임워크의 지적 핵심.

| GEO 원칙 (geo-for-ai-visibility) | 게이트 룰 | 판정 |
|---|---|---|
| 인용·통계·출처가 가시성 +30~41% | `min-sources` `numeric-claim-sourced` `citation-exists` | 실재(출처 수·무출처 수치 주장·URL 200+메타 일치): 결정적 / 지지(출처가 주장을 실제로 뒷받침): REVIEW 위임 |
| 30일 내 갱신 글이 인용 76.4% | `freshness-queue` | 결정적 |
| lastmod 위조는 역효과 | `honest-lastmod` (본문 diff 없는 lastmod 변경 거부) | 결정적 |
| 훈련봇/검색봇 구분 대응 | `robots-policy-match` | 결정적 |
| llms.txt = 중요 콘텐츠 안내 | `llmstxt-sync` (발행 목록과 일치) | 결정적 |
| Schema.org JSON-LD | `jsonld-valid` | 결정적 |
| hreflang 다국어 권위 | `hreflang-complete` | 결정적 |
| 태그 일관성 = topical authority | `tag-taxonomy` `no-orphan-tag` | 결정적 |
| 내부 링크 = 토픽 클러스터 | `min-internal-links` `no-isolated-post` | 결정적 |
| Wayback 시점 증거 | `archive-receipt` | 영수증 |
| 엔티티 명확성·크로스플랫폼 일관성 | — 저자 엔티티 스키마(박준우 엔티티 작업 참조)로 부분화 | 부분 |
| 소셜(X·Reddit·HN) 브랜드 검색량 | 게이트화 부적합 — 리포트 지표로만 | 측정만 |

근거 판정의 등급 분리: 구 `claim-evidence`는 **실재**와 **지지**로 쪼갠다. 실재(출처가 충분히 있고, 수치 주장에 출처 링크가 붙어 있고, 인용 URL이 실제로 살아 있으며 제목이 인용 표기와 겹침)는 위 3룰이 결정적으로 자동 판정한다. 지지(그 출처가 그 주장을 실제로 뒷받침하는가)는 비결정이라 룰이 아니다 — 퀘스트의 REVIEW 산출물(검토 기록 존재)로만 확인한다. `citation-exists`의 일시 오류(5xx/timeout)는 ERROR가 아닌 RETRY 진단이며, 검증 결과는 영수증 파일(`.abloq/citation-receipts.json`)에 24h 캐시된다.

## 6. 측정 설계 — 가시성 파이프라인 (백엔드)

제일 어려운 조각. AI 인용은 직접 관측이 안 되므로 **프록시 3계층**으로 설계한다. 전부 결정적 코드(§3.3 백엔드 모듈)다 — 에이전트가 개입할 자리가 없다.

1. **크롤 계층 (이미 보유, 결정적):** CloudFront 로그에서 AI봇 UA(GPTBot, ClaudeBot, OAI-SearchBot, Claude-SearchBot, PerplexityBot, ChatGPT-User, Claude-User…) 히트를 글별·봇별 집계. analyze-stats.py 확장으로 즉시 구현 가능. *훈련봇 히트 = 학습 유입, fetcher 히트 = 실시간 인용 발생의 직접 증거* — User-triggered fetcher(ChatGPT-User 등)의 히트는 사실상 "지금 누군가의 대화에 내 글이 인용됨"이다. **이게 우리만 쉽게 가진 신호다** (대시보드 SaaS는 고객 CDN 로그가 없다).
2. **색인 계층 (API, 결정적):** GSC 노출·클릭 추이, 색인 상태. 이미 SA 연동 완료.
3. **인용 계층 (샘플링, 비결정적):** 표준 질의 셋(글 주제별 N개)을 주기 실행해 ChatGPT/Perplexity/AI Overview 응답 내 인용 여부를 기록. 판정이 아니라 **추세 기록**으로만 취급 — 파이프라인은 실행과 기록까지만 책임지고, 결과는 큐의 우선순위 가중치로만 쓴다(결과 자체를 게이트화하면 비결정성이 룰을 오염시킨다). 질의 셋의 최초 작성만 에이전트 보조(글 주제에서 질의 도출), 이후 실행은 코드.

리포트 산출물: 글별 [크롤 히트 × 색인 추이 × 인용 샘플] → 스캐너(§3.3)의 우선순위 가중치 → 갱신(2)·근거 보강(3)·클러스터 큐레이션(4) 퀘스트의 큐 입력. **측정이 다음 노동을 지정하는 구조 = 운용 루프의 래칫.**

## 9. 리스크

- **일반화의 늪:** 테마 요청·엣지 케이스 지원이 늪. 방어: blog.yaml에서 파생되는 것만 지원, 레이아웃은 "교체 가능하나 지원 안 함" 선언.
- **백엔드가 더하는 운영 복잡도:** 정적 사이트의 매력은 서버 없음인데 백엔드(+PostgreSQL)가 그걸 깬다. 방어: 전 모듈 CLI 단독 실행 가능(§3.3), 백엔드는 스케줄·상태가 필요한 사용자만 켜는 옵션(docker-compose 한 장).
- **llms.txt 표준 불확실성:** 채택 실패 가능. 방어: 자동 생성이라 비용 미미, .md 병행 서빙이 본질이고 llms.txt는 입구일 뿐.
- **인용 측정의 비결정성:** §6에서 게이트는 절차까지만 판정하는 것으로 격리 완료.
- **GEO 자체의 평판 리스크:** "AI 조작" 프레임. 방어: 위조 차단 룰(honest-lastmod, 실재 출처 검증)을 전면에 — *우리는 인용받을 자격을 기계로 증명하는 쪽* 이라는 서사.
