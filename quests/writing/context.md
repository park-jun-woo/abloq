# 집필 퀘스트 — 컨텍스트 규약

집필 에이전트에게 주입되는 공통 규약. 게이트가 기계 판정하는 것과 규약으로만 묶는 것을
구분해 적는다 — 규약 위반 중 결정 검사가 가능한 것은 전부 게이트 룰이 잡는다.

## 출처 표기 형식

- 본문 인라인: `[표기 텍스트](https://...)` — 표기 텍스트는 인용한 페이지의 제목과
  토큰이 겹쳐야 한다 (citation-exists가 title/og:title 대조).
- 출처 섹션(`structure.headings.sources`): 글에서 인용한 출처를 목록으로 나열한다.
  `geo.min_sources` 이상이어야 한다 (min-sources).
- 수치 주장(숫자+단위+단정)은 같은 문단에 출처 링크가 있어야 한다 (numeric-claim-sourced).

## 작업 로그 (worklog)

- T1 수집 기록(claim ID → URL → 인용 예정 문구), T2~T3에서의 결정 사항을 시간 순으로 적는다.
- 로그는 REVIEW 검토자의 입력 자료다 — 게이트는 내용을 판정하지 않지만, 없으면 검토자가
  대응 근거를 추적할 수 없다.

## changelog

- 기존 글을 수정하는 작업이라면 글의 changelog 섹션(블로그 구조에 선언된 경우)에 변경
  내역 한 줄을 추가한다. 신규 글은 해당 없음.

## 금지 사항 (치즈 방어)

1. **출처 날조 금지** — 존재하지 않는 URL, 제목과 무관한 표기를 만들지 마라.
   citation-exists가 실 HTTP로 검증한다. 날조는 FAIL Fact로 돌아온다.
2. **범위 밖 파일 변경 금지** — 이 퀘스트의 산출물은 대상 글 1편 + 작업 로그 +
   REVIEW 기록뿐이다. blog.yaml·다른 글·레이아웃을 건드리지 마라.
3. **front matter 보존** — 기존 글 수정 시 front matter는 lastmod 갱신 외에 바꾸지
   마라 (front-matter-intact). 신규 글은 front-matter-schema 필수 필드를 채워라.
4. **자가 REVIEW 금지** — 집필 컨텍스트가 REVIEW 기록을 작성하면 안 된다. REVIEW는
   반드시 별도 컨텍스트(다른 세션/프로세스)의 검토자가 작성하고, 기록의 `reviewer:`
   식별자로 그 분리를 남긴다. 기록 날조의 완전 차단은 비결정이라 불가능하다 — 이 규약과
   기록 형식(검토 컨텍스트 식별자 + claim별 disposition)이 감사 가능성을 만든다.
5. **게이트 우회 금지** — 게이트는 Base 없는 신규 판정(전량 신규)으로 돈다. git 작업
   트리 상태로 근거 룰을 침묵시키는 우회는 막혀 있다.
