# 번역 퀘스트 — 태스크 트리 (T1~T3)

이 트리는 래칫 상태가 아니다. 래칫(게이트)은 최종 제출물(번역 파일 1편)만 판정한다.
아이템은 글 × 언어 — 언어마다 독립이라 한 언어의 FAIL이 다른 언어의 PASS에 영향을
주지 않는다.

## T1 — 원문 분석

- 원문의 구조 목록을 뽑는다: 헤딩 레벨 시퀀스, 문단 블록 수(빈 줄 경계), 이미지 경로,
  코드블록, 외부 링크 URL, 내부 글 링크 — translation-parity가 이 전부를 원문과
  양방향 비교한다.
- 내부 글 링크마다 대상 언어 치환 계획을 세운다: `/section/slug/` →
  `/{대상언어}/section/slug/` (게이트가 자기 언어 프리픽스를 강제한다).
- 인식 섹션 헤딩의 대상 언어 텍스트를 프롬프트의 heading map에서 확인한다.

## T2 — 번역

- front matter: `title`은 번역, `slug`는 원문과 동일(또는 둘 다 미지정 = 파일 어간 동일),
  `date`·`lastmod`는 **원문 값을 그대로 이식**(fm-mirror가 검사), `tags`는 원문 그대로
  유지(태그 어휘는 기본 언어 공유 — 클러스터 그래프가 기본 언어 태그를 쓴다).
- 인식 섹션 헤딩은 blog.yaml `structure.headings`의 대상 언어 텍스트를 정확히 쓴다.
- 코드블록·이미지 경로·외부 링크 URL은 한 글자도 바꾸지 않는다 (context.md 불변 목록).
- 문단 분할(빈 줄 경계)을 1:1로 보존한다 — CJK·RTL에도 적용된다.

## T3 — 자기 점검과 제출

- T1의 구조 목록과 번역본을 대조한다: 섹션 수·순서, 문단 수, 이미지/코드/링크 multiset,
  내부 링크 프리픽스, date·lastmod 이식.
- (선택) 의미 보존 REVIEW: 의미 동등성(오역·누락·과잉 의역)은 비결정이라 게이트 룰이
  아니다. 원하면 별도 컨텍스트 검토자에게 원문·번역 쌍을 제시해 소견을 받아라 —
  기록 강제 없음 (§10.2 미해결, 게이트는 구조 보존까지만 잠근다).

## 제출

`abloq quest translation submit --key <key> --in <submission.json>` — submission.json:

```json
{
  "article": "content/<lang>/<section>/<slug>.md"
}
```

경로는 블로그 루트(blog.yaml 위치) 기준 상대 경로이고 시드된 대상 경로와 일치해야
한다. FAIL이면 Fact(위치·기대·실제)가 돌아온다 — 수정 후 재제출.
**MaxTries=3**: FAIL 3회 누적 시 DONE으로 영구 잠금되니 Fact를 정확히 반영해라.
